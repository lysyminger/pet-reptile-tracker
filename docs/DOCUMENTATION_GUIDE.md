# 文档管理规范

## 📁 文件夹命名规则

每次操作/功能开发创建一个独立文件夹，命名格式：

```
docs/YYYY-MM-DD-功能名称/
```

### 示例
- `2026-03-27-delete-feature/` - 删除功能
- `2026-03-26-login-fix/` - 登录修复
- `2026-03-25-pet-avatar/` - 宠物头像功能

---

## 📄 文件夹内容

每个操作文件夹应包含：

### 必需文件
- `README.md` - 本次操作的概览和修改清单

### 可选文件
- 详细功能说明文档
- 截图/示意图
- 测试报告
- 用户指南

---

## 📝 README.md 模板

```markdown
# 操作文档 - [功能名称]

**日期：** YYYY-MM-DD  
**操作人：** [开发者]  
**功能：** [简短描述]

---

## 📋 本次修改内容

### 修改的文件列表
- `path/to/file1.ext` - 修改说明
- `path/to/file2.ext` - 修改说明

---

## 🎯 功能说明

[功能详细描述]

---

## 📁 文件结构

[项目结构树]

---

## ✅ 测试清单

- [ ] 测试项 1
- [ ] 测试项 2

---

*文档创建时间：YYYY-MM-DD HH:mm*
```

---

## 🗂️ 根目录文档

根目录只保留**长期有效**的文档：

### 保留
- `README.md` - 项目说明
- `QUICKSTART.md` - 快速开始指南

### 不保留
- 临时修改说明
- 功能开发文档
- 修复日志

这些都应该放入 `docs/日期 - 功能/` 文件夹中。

---

## 📦 归档

当某个功能的文档不再需要频繁访问时，可以：
1. 保留在 `docs/` 中（默认）
2. 压缩归档到 `docs-archive/`（可选）
